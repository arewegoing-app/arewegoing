# Security policy

> Found a security issue? Report it privately, not in a public issue. Details below.

## How to report

- **Preferred:** open a private GitHub Security Advisory at https://github.com/olitreadwell/arewegoing/security/advisories/new
- **Fallback:** email oliver.treadwell@gmail.com with `[arewegoing security]` in the subject

Include what you found, how to reproduce it, and what impact you think it has. A short proof-of-concept is fine. PGP is not set up yet; if you need encrypted comms, say so in the first message and we'll arrange a channel.

You should get a human reply within 72 hours.

## Scope

In scope:

- The production web app at the live deployment URL
- Email-driven RSVP and pledge endpoints (the token-gated routes under `/api/rsvp` and `/api/pledge`)
- HMAC token signing and verification (`lib/tokens/`)
- The inbound email webhook (`/api/inbound/email`) and its bearer auth
- Auth flows (NextAuth + Google OAuth, dev credentials in non-prod)
- Secret handling in deploys (env var leaks, log spillage, exposed `.env*` files)

Things we treat as in-scope vulnerabilities:

- Token forgery or replay against RSVP, pledge, or unsubscribe links
- Auth bypass or session fixation
- RSVP or pledge tampering on someone else's behalf
- Inbound webhook abuse (auth bypass, injection through email payloads)
- Secret leak through logs, build output, or deployment metadata
- SSRF, SQLi, stored XSS, CSRF on state-changing endpoints
- IDOR on event, recipient, or purchase records

## Out of scope

- Social engineering of maintainers or contributors
- Physical attacks, lost devices, shoulder-surfing
- Third-party dependency CVEs we have not shipped a fix for upstream of (report those to the dep maintainer first; tell us if it affects us in production)
- Missing rate limits on endpoints we already know about (see the `INBOUND_AUTH_OFF` and dev-mode notes in `README.md`)
- Reports generated by automated scanners with no human triage
- Best-practice findings without a concrete impact (e.g. "header X is missing" without showing what an attacker can do with that)
- Anything against non-production preview deployments unless it also reproduces in prod

## Disclosure

- 90 days from triage by default
- Sooner if a fix ships sooner, with credit to the reporter
- Longer only if we ask and you agree

If a fix needs coordination (third-party dep, infra), we'll tell you and propose a date.

## Credit and bounty

No paid bounty. This is a tiny coop with no security budget yet.

Public credit in the release notes and the security advisory is yours if you want it. Tell us the name or handle to use, or ask to stay anonymous.

## Once we know

When a report lands, we will:

1. Acknowledge within 72 hours
2. Confirm or reject the finding, with reasoning
3. Patch in a private branch, test, and ship
4. Publish the advisory after the fix is live
5. Credit the reporter (if they want credit)
